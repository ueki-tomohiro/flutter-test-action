# Flutter Test Report

Create a GitHub check run and pull request comment from Flutter test output.

This action reads:

- `flutter test --machine` JSON output
- `flutter test --coverage` LCOV output

It then publishes:

- a check run with a summary table and per-test details
- a pull request comment with test and coverage totals
- failure annotations for failed tests, up to GitHub's 50-annotation limit

## What It Expects

### Test report input

Generate machine-readable test output:

```sh
flutter test --machine > test.json
```

### Coverage input

Generate LCOV coverage output:

```sh
flutter test --coverage
```

This creates `coverage/lcov.info` by default.

## Inputs

| Name | Required | Default | Description |
| ---- | ---- | ---- | ---- |
| `machinePath` | no | - | Path to the file generated by `flutter test --machine` |
| `coveragePath` | no | - | Path to the LCOV file, typically `coverage/lcov.info` |
| `title` | no | `Flutter test results` | Title used for the GitHub check run |
| `token` | no | `${{ github.token }}` | GitHub token used to create the check run and PR comment |

You can pass only `machinePath`, only `coveragePath`, or both.

## Example Workflow

This action is designed primarily for `pull_request` workflows, because it posts a pull request comment in addition to creating a check run. On non-`pull_request` events such as `push`, it skips the comment and only creates the check run.

```yaml
name: flutter-test-report

on:
  pull_request:

permissions:
  contents: read
  checks: write
  issues: write

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v6

      - uses: subosito/flutter-action@v2
        with:
          channel: stable

      - run: flutter pub get

      - name: Run Flutter tests
        run: flutter test --machine > test.json

      - name: Run Flutter coverage
        run: flutter test --coverage
        if: success() || failure()

      - name: Publish test report
        uses: ueki-tomohiro/flutter-test-action@v1
        if: success() || failure()
        with:
          machinePath: test.json
          coveragePath: coverage/lcov.info
```

## Behavior

- If `machinePath` is provided, the action creates a unit test summary and detailed test table.
- If `coveragePath` is provided, the action adds file-level coverage summaries and overall hit/found totals.
- The check run conclusion is based on the test report status when test data is present.
- Coverage is considered successful when total line coverage is greater than `80%`.
- Very large titles, summaries, and details are truncated to GitHub's API limits.
- Only the first 50 annotations are sent, because GitHub check runs cap annotations per request.

## Notes

- On pull requests from forks, `GITHUB_TOKEN` is often read-only. In that case, GitHub may block the comment or check creation unless repository settings allow write access for forked PR workflows.
- On non-`pull_request` events, the action skips the PR comment because there is no issue/PR number in the workflow context.
- Because the action uses `issues.createComment` on pull requests, it is best used on `pull_request` events when you want the comment as well as the check run.
- This repository commits `dist/` on purpose. GitHub Actions execute the checked-in bundle referenced by `action.yml`.

## Development

This repository uses `mise`, `pnpm`, `oxlint`, `oxfmt`, `vitest`, and `lefthook`.

```sh
mise install
pnpm install
pnpm all
```

`pnpm install` runs `lefthook install`, so:

- `pre-commit` formats and lints staged TypeScript files
- `pre-push` runs the test suite

When you change runtime code, regenerate the committed bundle:

```sh
pnpm package
```
